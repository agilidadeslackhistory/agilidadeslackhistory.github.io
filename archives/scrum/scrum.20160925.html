sergio.giraldo 13:53 >> um backlog é um artefato que se renova, você encontrará isto descrito na literatura como _grooming_. usualmente o seu backlog começa com épicos, estes épicos são quebrados em histórias, estas histórias são quebradas em histórias menores e que podem ser executadas dentro das suas iterações. este processo de renovação e quebra é feito a partir do conhecimento que se adquire sobre os objetivos e este conhecimento se traduz em novas estimativas, as quais se refletem no seu plano de entrega. este detalhamento acontece durante todo o seu projeto, o PO ajuda a detalhar as histórias a partir da interação com os stakeholders, os devs colocam maiores detalhes a partir do que codificam e da arquitetura sendo desenvolvida, os analistas contribuem a partir das conversas com os usuários, etc Esta é uma das razões pelas quais é usual ter um release backlog com o roadmap do seu projeto ao longo das sprints, i.e., quais épicos o time pretende entregar. Este backlog contém uma ideia de entregas que está alinhada com a estratégia do produto/da organização e, por esta razão (estratégica), recomenda-se que este plano seja revisado a cada <n> sprints (eu faço a cada término de sprint).
